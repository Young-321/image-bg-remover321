## 🚀 标准部署流程（4步）

### 第一步：代码提交（如需要）
```bash
git add .

git push
```

### 第二步：构建项目
```bash
npm run build
```
**验证**: 确保 `out/` 目录生成成功

### 第三步：执行部署脚本
```bash
./deploy-global-key-direct.sh
```

### 第四步：验证部署
- 访问部署成功后显示的URL
- 等待1-2分钟让DNS生效
- 测试核心功能

## 🔧 部署脚本详解

### deploy-global-key-direct.sh 工作原理

```bash
#!/bin/bash
# 核心步骤：

# 1. 配置环境变量
export CLOUDFLARE_EMAIL="yangyong900829@gmail.com"
export CLOUDFLARE_API_KEY="cfk_6WREwnbWbd9PG3sLm91Udh5EMjqpdj2vd52p6SD725bcdff5"
export CLOUDFLARE_ACCOUNT_ID="765b6d5d0026cb618e703a30fe28f383"

# 2. 创建wrangler配置文件
cat > ~/.wrangler/config/default.toml << 'EOF'
[cloudflare]
email = "yangyong900829@gmail.com"
api_key = "cfk_6WREwnbWbd9PG3sLm91Udh5EMjqpdj2vd52p6SD725bcdff5"
account_id = "765b6d5d0026cb618e703a30fe28f383"
EOF

# 3. 执行部署
npx wrangler pages deploy out --project-name=bg-remover
```

## ⚠️ 已淘汰的部署方式

### ❌ 不再使用的脚本
- `deploy-now.sh` - API Token权限问题
- `deploy-with-token.sh` - API Token过期
- `deploy-with-global-token.sh` - Token问题
- `deploy-with-global-key.sh` - 配置问题
- `deploy-pages.sh` - 配置不完整
- `auto-deploy.sh` - 过于复杂
- `full-auto-deploy.sh` - 过于复杂
- `final-automated-test.sh` - 测试脚本
- `final-deploy-and-test.sh` - 过于复杂

### ❌ 不再使用的配置
- GitHub Actions自动部署 - 不稳定
- 专用API Token部署 - 权限管理复杂
- 多步骤复杂脚本 - 容易出错

---

## ✅ 成功经验总结

### 关键成功因素
1. **使用Global API Key而非专用Token** - 更稳定，权限更完整
2. **直接配置wrangler配置文件** - `~/.wrangler/config/default.toml`
3. **先构建后部署** - 确保 `out/` 目录存在
4. **单一职责脚本** - 只做部署一件事，简单可靠

### 避免的问题
- ❌ 不要依赖GitHub自动部署
- ❌ 不要使用过期的API Token
- ❌ 不要使用过于复杂的脚本
- ❌ 不要跳过手动验证步骤
